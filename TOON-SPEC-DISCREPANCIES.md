# TOON conformance discrepancies: spec vs. reference implementation vs. yatl

This document records known, unresolved differences between three things:

1. the formal [TOON specification](https://github.com/toon-format/spec),
2. the [reference implementation](https://github.com/toon-format/toon), and
3. this project (**yatl**).

Some rows are cases where yatl is stricter than the spec, some are cases where
the reference implementation is more lenient than the spec, and some are genuine
spec ambiguities where the two implementations diverge. None of them is resolved
yet; the table is a working record, not a conformance guarantee.

**Source:** the comparison below was produced by **Fable 5**.

**As of:** 2026-07-02, yatl commit
`1a13f902890bc7bcf2e290d3741af276e6d670e8`.

| topic | spec | reference implementation | yatl | assessment |
|-------|------|--------------------------|------|------------|
| Token trimming around split tokens (inline array / tabular cells) | §11.2 and Appendix B.3: surrounding "spaces" are trimmed; where precise the spec always means ASCII space U+0020 (§12 says 'exactly one ASCII space (U+0020)'). 'Whitespace' is used loosely in §9.1/§12 but never defined. | Uses JavaScript String.prototype.trim(), which strips the full Unicode whitespace set: TAB, NBSP U+00A0, U+2000-200A, U+3000, U+FEFF, etc. Verified: 'x[2]: a,\<NBSP\>b' -> ["a","b"]; 'x[2]: a,\<TAB\>b' -> ["a","b"]. | trim_spaces() strips U+0020 only. Verified: NBSP is preserved in the decoded token (' b'); a TAB-padded bare token is a strict-mode error (unquoted value requires quoting per §7.2). | yatl matches the normative text; reference over-trims, which can silently alter data (a bare token legally starting with NBSP loses it). Worth filing a spec issue: 'whitespace' is undefined and the reference impl contradicts B.3. |
| \uXXXX surrogate pairs for supplementary characters | §7.1: supplementary scalar values (U+10000-U+10FFFF) MUST be emitted as literal UTF-8 and surrogate \uXXXX escapes MUST be rejected by decoders; U+D800-DFFF from \uXXXX MUST be rejected. | Rejects both lone surrogates and surrogate pairs: 'x: "😀"' -> error 'lone surrogate; supplementary code points MUST appear as literal UTF-8'. Conforms. | Accepts the pair and combines it JSON-style into U+1F600 (verified: decodes to the emoji). Only lone surrogates are rejected. | yatl violates §7.1 (JSON-style leniency). If strict conformance is the goal, yatl should reject paired surrogate escapes too - candidate fix. |
| Duplicate sibling object keys | §14.4: with strict=true (default) decoders MUST error; non-strict MUST apply last-write-wins silently. | Errors in strict mode: 'a: 1\na: 2' -> 'Duplicate sibling key "a"'. Conforms. | No duplicate detection at any nesting level: emits both key events and parses successfully (verified at root and nested). | yatl deviates from §14.4. Defensible for a SAX API (YAJL behaves the same; the consumer sees every event), but a conformant strict mode needs per-scope key tracking or a documented deviation. |
| Indentation not an exact multiple of indentSize | §12/§14.2: in strict mode the number of leading spaces MUST be an exact multiple of indentSize (default 2); MUST error otherwise. | Errors: 'a:\n   b: 1' (3 spaces) -> 'Indentation must be exact multiple of 2'. Conforms. | Accepts 3-space indentation and decodes {a:{b:1}} (verified). yatl does not validate the multiple-of-indentSize invariant. | yatl violates a strict-mode MUST. Candidate fix in yatl's indent handling. |
| Over-indented list item (depth jump below array header) | §9.2/§9.4: list items appear at depth +1 under the header. An item at depth +2 is not an item of that array. | Treats a +2-indented '- a' as outside the array: 'x[1]:\n    - a' -> error 'Expected 1 list array items, but got 0'. Conforms. | Accepts the over-indented line as a list item and decodes {x:["a"]} (verified). | Same root cause as the indentation row: yatl's depth checks are lenient. yatl deviates from spec. |
| Bare keys that do not match the key grammar (e.g. 'a b: 1', 'a : 1') | §7.3: keys MAY be unquoted only if matching ^[A-Za-z_][A-Za-z0-9_.]*$ - an encoder-side MUST-quote rule. Decoder-side handling of nonconforming bare keys is not in the §14 strict-error catalogue; §12 says surrounding whitespace around tokens SHOULD be tolerated. | Lenient: any text before the first colon is the key. 'a b: 1' -> {"a b":1}; 'a : 1' -> {"a":1}. | Rejects both with 'unquoted value requires quoting' (the line fails key parsing and the fallback scalar contains a colon). | Spec is ambiguous (silent on decoder behavior). The two implementations genuinely diverge; interop hazard for hand-written documents. Worth a spec clarification. |
| Unquoted value containing a colon ('x: a:b') | §7.2/§15: encoders MUST quote strings containing colons. §4: decoder-side, unquoted tokens fall through to 'otherwise -> string'. §14 does not list this as a strict-mode error. | Decodes as the string "a:b" (post-colon remainder taken as one token). | Strict-mode error: 'unquoted value requires quoting' (rejects any bare token a conforming encoder would have quoted; opt-out via yatl_lenient_scalars). | Divergence rooted in spec ambiguity: yatl enforces encoder-side quoting rules when decoding; the reference does not. Both readings are defensible; interop differs on malformed-but-recoverable input. |
| '- []' as a list item | §9.2: empty inner arrays in list position are '- [0\<delim?\>]:'; the '[]' form explicitly does NOT apply to list-item inner arrays ('key: []' and root '[]' are the only [] positions, §9.1). | Accepts it: 'x[1]:\n  - []' -> {"x":[[]]}. | Rejects it ('malformed array length in header') after the applied commit be2edfe restricted [] to root and key-colon positions. | yatl matches the spec text; the reference is lenient beyond the spec. Another candidate upstream issue. |
| Bare line 'key[]' (bracket segment, no length, no colon) | §6 requires every array header to have a length and a colon; §4 says a key MUST be followed by a colon; but §5 root-form discovery says a single line that is neither a header nor a key-value line decodes as a root primitive string. | Falls through to root primitive: 'key[]' -> the string "key[]". | Parse error ('malformed array length in header'). | Divergence with a real spec tension: per §5 the reference's answer is arguable, but a bare token containing brackets must be quoted per §7.2, which supports yatl's rejection. Needs spec clarification. |
| Numeric overflow (e.g. 'x: 1e309') | §4: if a numeric token is not representable, implementations MAY return a higher-precision type, a string, or a documented approximation; erroring is not among the listed options; policy MUST be documented. | Returns the original token as a string: {"x":"1e309"} (silent type change from number to string). | Strict parse error: 'numeric (floating point) overflow'. | Both are edge readings: the reference's string fallback is sanctioned but silently changes the value's type; yatl's hard error is predictable but not one of the spec's listed MAY options. Divergence worth documenting in YAJL-DIFFERENCES.md / spec issue. |
| Integer precision beyond 2^53 | §4: lossless-first is RECOMMENDED for data-interchange libraries; approximation allowed only as documented policy. | IEEE-754 double only: 9007199254740993 decodes to 9007199254740992 - silent precision loss. | Lossless 64-bit integer callback: decodes 9007199254740993 exactly. | yatl follows the RECOMMENDED behavior; the reference silently corrupts integers above 2^53. Interop hazard when round-tripping IDs through the reference impl. |
| Invalid UTF-8 input bytes | §17: charset is always UTF-8. §15: decoders SHOULD accept valid UTF-8; the spec never says what to do with invalid byte sequences. | Cannot see raw bytes: decode() takes a JS string, so byte validation happens (or silently does not) at the caller's I/O layer, typically replacing bad sequences with U+FFFD. | Validates UTF-8 in strings/keys by default and raises 'invalid UTF-8 in string' (verified); disable via yatl_dont_validate_strings. | Different validation posture at the API boundary. yatl is stricter than both the spec's SHOULD and the reference; the reference can silently launder mojibake into a document. |
| Number type surfaced to the host (informative) | Spec maps tokens to host numbers; it does not mandate an integer/float distinction. | Single JS number type: '1e6' -> 1000000 (indistinguishable from integer 1000000). | SAX callbacks distinguish integer vs double: '1e6' -> double 1e+06; '42' -> integer 42. | Not a conformance issue - a host-mapping difference to keep in mind when comparing outputs. |
